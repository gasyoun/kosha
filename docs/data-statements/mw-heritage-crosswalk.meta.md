# Data statement — MW ↔ Heritage entry-level crosswalk (`mw-heritage-crosswalk`)

_Created: 11-07-2026 · Last updated: 11-07-2026_

Data statement for the `mw-heritage-crosswalk` dataset served by the kosha
data-hub. Manifest row:
[data/manifest/datasets.json](https://github.com/gasyoun/kosha/blob/main/data/manifest/datasets.json).
Download: [`mw_heritage_crosswalk.tsv` in release data-v0.1.0](https://github.com/gasyoun/kosha/releases/tag/data-v0.1.0).

## Composition & counts

185,803 rows, one per MW headword key, mapping into the Sanskrit Heritage
Platform (INRIA, Gérard Huet) dictionary hypertext. 25,140 keys are covered by
a Heritage entry; of those, 97.6% resolve to a concrete entry anchor. TSV,
UTF-8, 3 columns:

| Column | Content |
|---|---|
| `mw_key1` | MW headword, normalized key1 (SLP1) |
| `covered_flag` | 1 if the headword exists in Heritage's DICO, else 0 |
| `heritage_entry_anchor` | Heritage hypertext anchor (deep-link target) where resolved |

The full-lexicon shape is deliberate: uncovered headwords are present with
`covered_flag = 0`, so coverage statistics are computable from the file alone.

## Source provenance

Built by the H099 Heritage-mirror pipeline in
[gasyoun/SanskritLexicography](https://github.com/gasyoun/SanskritLexicography)
(`HeadwordLists/mw_heritage_crosswalk.tsv`): a sparse local mirror of the
Heritage Platform resources (via the darkone23/Heritage_Resources GitHub
mirror) was scanned for entry anchors, and MW key1 headwords were matched
through a validated Velthuis↔SLP1 transliteration bridge.

## Curation rationale

The Heritage Platform is the strongest morphology-first Sanskrit resource;
MW is the reference bilingual dictionary. Linking them entry-to-entry lets a
dictionary UI offer "open this word in Heritage" deep links and lets coverage
studies quantify where the two traditions diverge. The *crosswalk* (key pairs
+ anchors) is our own derived work and publishable; it was built precisely so
the link layer could be public while the mirrored Heritage content itself
stays restricted.

## Language variety

Sanskrit (ISO 639-3 `san`), Classical register as covered by MW and Heritage's
DICO. Keys in SLP1 (key1 normalization); Heritage's native Velthuis encoding
is bridged, not exposed — anchor strings follow Heritage's own conventions.

## Annotator / process information

Programmatic matching (transliteration bridge + anchor scan); no LLM-generated
content; no manual per-row annotation. The 2.4% of covered keys without a
resolved anchor are match-without-address cases left unresolved rather than
guessed.

## Known biases & limitations

- Heritage's DICO is compact (~tens of thousands of entries) against MW's
  ~186k keys, so `covered_flag = 0` dominates by design — the file measures
  Heritage's selective scope, not data loss.
- Matching is at the normalized-key level: homonym alignment between MW
  numbering and Heritage's entry splitting is not sense-verified.
- Anchors are valid against the mirrored Heritage snapshot (H099); upstream
  Heritage releases can move anchors. Treat anchor validity as
  snapshot-relative.
- **Rights boundary:** this file is publishable, but the Heritage mirror
  behind it is LGPLLR-pending and sits in the restricted tier
  (`heritage-mirror` manifest row). Do not bundle mirror content with this
  crosswalk.

## Intended use / known misuse

**For:** deep-linking dictionary UIs into Heritage (kosha's
`/api/v1/lemma` `heritage` witness, csl-atlas witness column), MW↔Heritage
coverage studies, transliteration-bridge validation. **Misuse:** shipping
Heritage DICO content alongside the anchors (rights boundary above);
interpreting `covered_flag = 0` as "word unknown to INRIA tooling" (Heritage's
morphology covers far more forms than its DICO lists as entries); relying on
anchors against a live Heritage build without re-validation.

## Maintenance & sunset plan

Regenerated by the H099 pipeline when the Heritage mirror snapshot is
refreshed or MW keys change; kosha re-mirrors at each `data-v*` cut. The
pending LGPLLR ruling on the mirror (`@DECIDE`, tracked in the manifest's
`heritage-mirror` row) does not gate this file, but a ruling that permits
mirror publication would likely trigger a richer successor (anchors + glosses),
flipping this row to `superseded by [X]`.
