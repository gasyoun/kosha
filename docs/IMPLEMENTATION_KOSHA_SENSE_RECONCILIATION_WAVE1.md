# IMPLEMENTATION — kosha sense-reconciliation, wave 1 (file-level)

Index: [PLAN_KOSHA_SENSE_RECONCILIATION_2026H2.md](https://github.com/gasyoun/kosha/blob/main/docs/PLAN_KOSHA_SENSE_RECONCILIATION_2026H2.md).
Architecture: [ARCHITECTURE_KOSHA_SENSE_RECONCILIATION.md](https://github.com/gasyoun/kosha/blob/main/docs/ARCHITECTURE_KOSHA_SENSE_RECONCILIATION.md).

Ordered build sequence. Each step names the files it touches and its dependency. Work in a git
worktree off `origin/main` (kosha is a guarded main-tree repo). Windows/encoding rules apply
(`sys.stdout.reconfigure(encoding="utf-8")`; `encoding="utf-8"` on output-capturing `subprocess`;
no UTF-8 BOM; multi-step logic in a `.py` file, not inline shell).

### Step 0 — pilot set (no dep)
- Pick ~500 polysemous headwords for the wave-1 pilot: headwords with ≥2 PWG leaf senses AND ≥1
  resolvable `<ls>` locus per sense AND non-trivial DCS attestation count. `nāgadanta`/`nāgadantaka`
  are the canonical smoke-test pair (must land a→tusk, b→peg).
- Write the list to `data/concordance/sense_pilot_headwords.tsv`. Emit a `LOG:` line with the
  selection query. **File:** new `scripts/select_sense_pilot.py`.

### Step 1 — consume the PWG sense→loci export (dep: H1456)
- Input `pwg_sense_loci.tsv` (contract in the Architecture doc). If absent, invoke `microstructure.py`
  directly per the H1456 spec to regenerate for the pilot headwords; if that genuinely fails, apply
  the Stop-condition (a) from the autonomy contract — do the gloss-overlap-only tier and park the rest.
- Load into a `{(slp1,hom): [Sense(sense_id, gloss_de, ls_loci[])]}` map. **File:** `scripts/build_sense_corpus_concordance.py` (new), loader section.

### Step 2 — `<ls>`-locus resolver (dep: step 1)
- For each sense's `<ls>` strings, resolve abbrev→source via `RussianTranslation/src/pwg_sources.py`
  (`pwgbib.txt`), then normalise the citation to the same address space as the DCS token loci using
  `concordance_core.citable_locus`. Output a `{sense → resolved_locus_set}` map.
- Record the resolution rate (resolved / total `<ls>` on pilot senses) — this is the wave-1 acceptance
  metric. **File:** `build_sense_corpus_concordance.py`, resolver section; helper reused from
  `concordance_core`.

### Step 3 — candidate senses per attestation (dep: step 2 + existing headword↔attestation link)
- Reuse `build_dict_corpus_concordance.py`'s headword→DCS-attestation table (do NOT re-query DCS from
  scratch — import its aggregation or read its `dict_corpus_concordance.tsv`).
- For each attestation of a pilot headword, generate candidates:
  - **(i) locus-match** — attestation's citable locus ∈ a sense's resolved locus set → `method=locus`, high conf.
  - **(ii) gloss/synset-overlap** — token overlap between the DCS `WordSem` gloss (or H1453's
    WordSem→MW-sense projection when present) and each sense's `gloss_de`/RU gloss → `method=overlap`, mid conf.
- **File:** `build_sense_corpus_concordance.py`, candidate section.

### Step 4 — LLM adjudication of the residue (dep: step 3)
- Rows with no candidate or ≥2 near-tied candidates → batch to a gloss-grounded LLM adjudicator via a
  **Claude Code Workflow** (`model:'sonnet'`; NO API key — the Workflow tool is the Max-subscription
  runner, per [project_pwg_ru]). Prompt: given the headword, the numbered senses with glosses, and the
  attestation's KWIC line, return `sense_id` + confidence + one-line reason. Never invent a sense id
  outside the provided set (hard guard — reject + re-ask on out-of-set).
- **File:** `build_sense_corpus_concordance.py` (residue batching) + `wf/sense_adjudicate.js` (new workflow script).

### Step 5 — assign + review queue (dep: steps 3–4)
- Assign the best-confidence sense per attestation. `confidence<τ` (default `τ=0.6`, a marked default —
  log it) → append to `sense_review_queue.tsv` (kept, not dropped). Emit
  `sense_corpus_concordance.tsv` (headword, sense_id, lemma, locus, conf, method, rights) and
  `sense_corpus_coverage.tsv`. Stamp every row's `rights` per `corpus_gate.RIGHTS`.
- **File:** `build_sense_corpus_concordance.py`, writer section.

### Step 6 — sense-sharded KWIC viewer (dep: step 5)
- Fork `concordance/dict/` → `concordance/senses/`; shard `kwic_<a>.js` keyed on `(headword, sense_id)`
  with ≤3 samples per (sense, source); index shows senses nested under the headword. Filter shards to
  `rights=public`. **Files:** `concordance/senses/index.html`, `concordance/senses/data/*.js`
  (generated by `build_sense_corpus_concordance.py --viewer`).

### Step 7 — report, manifest, publish gate (dep: steps 5–6)
- Write `data/concordance/SENSE_CONCORDANCE_BUILD_REPORT.md` (per-tier counts, locus-resolution rate,
  coverage, τ, `nāgadanta` worked example). Add `sense-corpus-concordance` row to
  `data/manifest/datasets.json`. Add a `CHANGELOG.md` `[Unreleased]` entry → the release is cut by
  `/cut-release`. Run `/publish-safety-check`; record the verdict in the PR. Only on GO does the public
  `concordance/senses/` viewer ship. **Files:** report, manifest, CHANGELOG.

### Step 8 — verify (dep: all)
